# 插件化执行系统 plugin-based-excutor

## 一、整体架构设计说明
- 进程内采用插件结构体形成的执行链，统一对插件进行管理、调用
- 进程自身作为 web 服务器，对外提供接口：插件执行、插件状态修改、插件信息获取
## 二、插件系统的核心设计思路
### 1. 插件加载
- 主进程从指定路径中，读取路径下所有的可执行文件插件信息，每个插件组装成一个 Plugin 结构体，多个结构体组成进程的插件执行链
### 2. 插件执行
- 主进程接收 web 请求，接收插件需要的 input 数据
- 拷贝当前的执行链的只读副本，依次对执行链上各个插件进行执行
- 将各个执行链的执行结果汇总，统一返回
### 3. 插件启用/禁用
- 对外提供接口，接收 web 请求，获取插件最新状态
- 拷贝当前的执行链的只读副本，修改插件的状态
- atomic.Pointer store 原子更新执行链
## 三、关键实现选择与取舍说明
- 采用 atomic.Pointer 对执行链进行存储，每次 Load 得到只读副本，Store 时原子性更新，防止出现更新的中间状态
- 提供 web 对外接口对插件进行管理

## 四、未完成部分的设计想法
### 1、插件热加载/热卸载机制
- 服务进程内起定时任务，周期记录插件配置文件、插件可执行文件的 sha256 值
- 若发生变动，证明插件发生修改，重新生成处理链
- 按当前新的重新生成处理链，替换旧插件执行链

### 2、插件执行超时控制
- 编写配置文件，统一对插件超时时间进行管理，包括：总超时时间，单个插件超时时间
- 服务进程启动，读取配置文件中各个插件的超时时间，放入执行链中
- 总任务启动时，根据超时时间创建 WithTimeout 上下文，超时后失败返回
- 每个插件启动时，根据超时时间创建 WithTimeout 上下文，超时后失败返回

### 3、插件隔离方案
- 将插件以容器方式运行，确保与宿主机上其他进程、资源的隔离性
- 环境安装轻量级的镜像，每次执行时，将插件需要的宿主机文件、插件输出路径等挂载进容器中
- 重写镜像启动命令为执行插件命令，使插件在容器中执行
- 读取宿主机上插件输出路径，获取插件执行结果

### 4、插件依赖关系与版本约束
- 运行时结果依赖：如果插件A依赖插件B的执行结果，编写配置文件，在配置文件中指定插件执行的顺序，基于此对插件执行链进行排序
- 运行时文件依赖、版本约束：针对每个插件，额外维护插件依赖关系树，在打包时交给本服务进程，由本服务进程进行依赖检查

### 5、插件执行结果的失败隔离与降级策略
- 失败隔离：编写插件异常处理，在失败时兜底调用进行处理
- 熔断策略：插件处理失败进行计数，一段周期内失败次数过多禁止再次调用，到达熔断时间后尝试调用，通过后正常，不通过继续触发熔断，防止持续失败
